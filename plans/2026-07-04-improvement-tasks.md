# copippe 改善タスク一覧(2026-07-04)

## この文書について

- **目的**: リリース済みプロダクト copippe(v0.3)の改善タスクを、AI コーディングエージェント(Codex 等)が設計・実装に着手できる詳細度で定義する。
- **前提となる現状判定**: copippe は「リリース済み・稼働中」。GitHub Releases に v0.1〜v0.3 が published(v0.3 は 2026-06-07、DMG/ZIP + SHA256 添付)、copippe.com(GitHub Pages、`docs/` がソース)で LP・ユーザーガイドを公開中。ソースに TODO/FIXME/スタブなし。テスト9ファイル(Swift Testing)あり。2026-06-11 の全面リファクタリング(`plans/2026-06-10-full-refactoring.md`)は実質完走済み。
- **作業ルール**: 実装前に必ずリポルートの `AGENTS.md` を読むこと。特に Red Team レビューのループ(収束まで最大10回)と、コミット対象を変更ファイルのみに限定するルールに従う。
- **ビルド・テスト**: Xcode プロジェクト(`copippe.xcodeproj`、単一ターゲット、Swift 6 / macOS 14+)。
  - ビルド: `xcodebuild -project copippe.xcodeproj -scheme copippe -configuration Release build`
  - テスト: `xcodebuild test -project copippe.xcodeproj -scheme copippe -destination 'platform=macOS'`
- **優先度**: P0 = 配布品質・信頼性(先にやる)、P1 = ユーザー価値の拡大、P2 = 中期改善。
- **オペレータ依存**: 一部タスクは Apple Developer Program 加入や GitHub 設定などユーザー(人間)の作業を含む。該当タスクに「オペレータ作業」として明記した。Codex はコード・設定ファイル側を先に整備し、オペレータ作業はチェックリストとして出力すること。

## アーキテクチャ前提(実装時に触る主要ファイル)

- `copippe/CopippeApp.swift` — `@main` エントリポイント + AppDelegate。MenuBarExtra 常駐。
- `copippe/ClipboardMonitor.swift` — NSPasteboard を 0.5 秒ポーリング監視。`convertToPlainText`(:81 付近)で RTF/HTML→プレーン変換、`extractImage` で画像取得。
- `copippe/HistoryManager.swift` — 履歴の保持・上限管理(既定30件・設定可)・永続化。
- `copippe/SnippetManager.swift` — スニペット CRUD・フォルダ整理。
- `copippe/HotkeyManager.swift` — Carbon ベースのグローバルホットキー登録・競合検出・永続化。
- `copippe/Persistence.swift` — atomic write の `JSONFileStore`。
- `copippe/ImageStore.swift` — 画像の PNG サムネイル保存・復元。
- UI: `PreferencesView.swift` + `GeneralTab.swift` / `HotkeyTab.swift` / `SnippetTab.swift`、検索ポップアップ `PopupContentView.swift`(Ctrl+Option+V、NSPanel)。
- テスト: `copippeTests/`(Swift Testing、本番データ非破壊の隔離ヘルパー `TestSupport.swift` あり)。

## P0: 配布品質・信頼性

### CP-01: GitHub Actions CI の導入(ビルド + テスト)

- **背景/現状**: `.github/` が存在せず CI が皆無。リファクタリング後の回帰検知が手元実行頼み。
- **作業内容**:
  1. `.github/workflows/ci.yml` を新規作成。トリガー: `push`(main)と `pull_request`。
  2. `macos-14` ランナー(または利用可能な最新 macOS ランナー)で `xcodebuild test -project copippe.xcodeproj -scheme copippe -destination 'platform=macOS'` を実行。Xcode バージョンは `xcode-select` で固定(Swift 6 対応版)。
  3. ビルド警告をログで確認できるよう `-resultBundlePath` を artifact 保存(任意)。
- **受け入れ基準**: PR 作成時に CI が走り、既存テスト9ファイルが全部 pass する。CI 上で失敗した場合に原因がログから追える。
- **検証**: ダミーブランチで意図的にテストを壊し、CI が fail することを確認してから戻す。
- **依存**: なし。最初に着手すべきタスク。

### CP-02: リリース自動化ワークフロー(tag → DMG/ZIP → GitHub Release)

- **背景/現状**: v0.1〜v0.3 は Xcode ローカルビルド + 手動アップロードで公開しており、再現性がない。リリースノートに SHA256 を手書きしている。
- **作業内容**:
  1. `.github/workflows/release.yml` を新規作成。トリガー: `v*` タグの push。
  2. ジョブ内容: `xcodebuild archive` → `.app` を取り出し → ZIP 生成(`ditto -c -k`)+ DMG 生成(`hdiutil` または `create-dmg`)→ 両ファイルの SHA256 算出 → `gh release create`(draft)に添付し、本文に SHA256 を自動記載。
  3. 署名は現状の ad-hoc のまま(CP-03 完了後に署名・公証ステップを差し込める構造にしておく。例: 署名ステップを別ジョブ/条件分岐に分離)。
  4. リリース手順書 `docs/RELEASING.md`(または `plans/` 配下)を作成: バージョン番号の付け方(semver)、タグ打ち → draft 確認 → publish の流れ、LP(`docs/index.html`)のダウンロードリンク更新手順。
- **受け入れ基準**: テスト用タグ(例 `v0.3.1-rc1`)を push すると draft release が自動生成され、DMG/ZIP/SHA256 が揃っている。draft のため公開はされない。
- **検証**: テストタグで実際に走らせ、成果物をローカルにダウンロードして起動確認。確認後テストタグと draft を削除。
- **依存**: CP-01(CI 基盤)。
- **注意**: release の publish 自体は必ず人間の操作とする(draft 止まり)。

### CP-03: Developer ID 署名 + Apple 公証(notarization)

- **背景/現状**: 現在 ad-hoc 署名(`DEVELOPMENT_TEAM` 空、`copippe.entitlements` は app-sandbox=false)。初回起動が Gatekeeper でブロックされ、README で `xattr -cr` 回避を案内している。配布上最大の摩擦であり、一般ユーザーの離脱要因。
- **オペレータ作業(前提)**: Apple Developer Program への加入(年額課金)、Developer ID Application 証明書の発行、App Store Connect API キーまたは Apple ID App 用パスワードの発行、GitHub Secrets への登録。**Codex はこの判断・購入を代行できない。加入するかはユーザー判断。**
- **作業内容(コード側、オペレータ作業と独立して先行可能)**:
  1. リリースワークフロー(CP-02)に署名・公証ステップを追加: 証明書 import(`security` コマンド or Apple 公式 action)→ `codesign --deep --options runtime` → `xcrun notarytool submit --wait` → `xcrun stapler staple`。Secrets 未設定時はステップをスキップして従来 ad-hoc ビルドにフォールバックする条件分岐にする。
  2. Hardened Runtime 有効化に伴う entitlements の見直し(クリップボード監視・グローバルホットキー・SMAppService は Hardened Runtime と共存可能だが、必要な例外があれば `copippe.entitlements` に追記)。
  3. 公証済み配布に切り替わった後の README / `docs/guide.html` / `docs/guide-ja.html` の Gatekeeper 回避手順の削除(公証完了後の別コミット)。
- **受け入れ基準**: Secrets 設定済み環境でリリースワークフローが公証済み DMG/ZIP を生成し、素の macOS で警告なく起動する。Secrets 未設定でもワークフローは fail しない。
- **検証**: `spctl -a -vv copippe.app` が accepted を返す。
- **依存**: CP-02。オペレータ作業完了までは「フォールバック付き実装まで」を完了条件とする。

### CP-04: 機密コピーの履歴除外(ConcealedType / TransientType 対応)

- **背景/現状**: パスワードマネージャ(1Password 等)がコピー時に付与する `org.nspasteboard.ConcealedType` / `org.nspasteboard.TransientType` を現状まったく見ていない(リポ内に該当文字列なし)。**パスワードが平文で履歴 JSON に永続化される**。クリップボードマネージャとして必須のプライバシー対応。
- **作業内容**:
  1. `ClipboardMonitor.swift` の取り込み判定に、pasteboard types に `org.nspasteboard.ConcealedType` または `org.nspasteboard.TransientType` が含まれる場合は履歴に記録しないガードを追加。`org.nspasteboard.AutoGeneratedType` の扱いも検討(既定: 記録しない)。
  2. 設定 UI(`GeneralTab.swift`)に「機密としてマークされたコピーを記録しない」トグルを追加(既定 ON)。
  3. テスト追加: `copippeTests/ClipboardMonitorTests.swift` に ConcealedType 付きペーストボード項目が履歴に入らないケース、トグル OFF 時は入るケース。
- **受け入れ基準**: 1Password 等からのコピーが履歴・永続化ファイルの双方に残らない。既存テスト全 pass。
- **検証**: ユニットテスト + 手動で実際のパスワードマネージャからコピーして履歴に出ないこと。
- **依存**: なし。CP-01 と並行可。

### CP-05: リファクタリング計画書の進捗記録の是正

- **背景/現状**: `plans/2026-06-10-full-refactoring.md`(2291行)の進捗表が「Phase 4.2〜8 未着手・push 未実施」のまま止まっているが、git 履歴上は同日中に Phase 8 まで完了・push 済み。ドキュメントと実態の乖離は後続エージェントの誤判断を招く。
- **作業内容**: 進捗表・チェックリストを実態(全 Phase 完了)に更新し、末尾に「2026-06-11 完了。以後の改善は plans/2026-07-04-improvement-tasks.md(本書)へ」と終結宣言を追記。計画本文は履歴として改変しない(進捗欄のみ更新)。
- **受け入れ基準**: 進捗表が git 履歴と一致し、文書冒頭または末尾でクローズ済みであることが分かる。
- **依存**: なし。最小タスク。

## P1: ユーザー価値の拡大

### CP-06: 履歴アイテムの個別削除

- **背景/現状**: リファクタリング時に YAGNI で削除された機能(計画書末尾の設計メモに記録あり)。「間違えてコピーした機密情報を1件だけ消したい」は定番ニーズで、CP-04 と補完関係。現状は全消去しかない(要確認: 全消去の有無も実装時に確認)。
- **作業内容**:
  1. `HistoryManager.swift` に `remove(id:)` を追加(永続化への反映含む)。画像アイテムの場合は `ImageStore.swift` のサムネイルファイルも削除。
  2. UI: メニューバーの履歴一覧と検索ポップアップ(`PopupContentView.swift`)の双方から削除できる操作を追加。macOS 慣習に合わせ、右クリックメニューまたはホバー時の削除ボタン + Delete キー。
  3. テスト: `HistoryManagerTests` に削除・永続化反映・画像ファイル掃除のケースを追加。
- **受け入れ基準**: 1件削除後、再起動しても該当項目が復活しない。画像の孤児ファイルが残らない。
- **依存**: なし。

### CP-07: ピン留め(お気に入り)機能

- **背景/現状**: ピン留め機能なし(リポ内に該当実装なし)。よく使う項目が履歴上限(既定30件)で押し出される。スニペット機能はあるが「登録」の手間があり、ワンタップのピン留めとは用途が異なる。
- **作業内容**:
  1. 履歴アイテムのモデルに `isPinned` を追加(永続化スキーマの後方互換に注意 — 既存 JSON に無いキーはデコード時に false 扱いにする)。
  2. ピン留め項目は上限カウントから除外し、一覧の先頭セクションに固定表示。
  3. UI: 履歴一覧・ポップアップからピン留め/解除。ピン留め項目からスニペットへの昇格導線(既存 `SnippetManager` へ登録)も任意で検討。
  4. テスト: 上限押し出し時にピン留め項目が残ること、永続化の互換性(旧フォーマット JSON の読み込み)。
- **受け入れ基準**: ピン留め項目が上限・再起動を越えて維持される。旧バージョンのデータファイルがそのまま読める。
- **依存**: CP-06 と同じファイル群を触るため、続けて実施するのが効率的。

### CP-08: 更新通知(軽量)→ 将来 Sparkle 自動更新

- **背景/現状**: 更新手段がなく、ユーザーは新バージョンに気づけない。フル自動更新(Sparkle)は署名・公証(CP-03)完了が事実上の前提。
- **作業内容(段階1: 軽量通知のみ、先行可)**:
  1. GitHub Releases API(`https://api.github.com/repos/<owner>/copippe/releases/latest`)を起動時 + 24時間毎に確認し、現行 `CFBundleShortVersionString` より新しければメニューバーに「アップデートあり」項目を表示 → クリックでリリースページを開く。
  2. 設定にチェック無効化トグル。ネットワーク失敗は黙って無視(オフライン動作を損なわない)。
  3. テスト: バージョン比較ロジック(semver 比較)のユニットテスト。
- **作業内容(段階2: Sparkle 2 導入、CP-03 完了後)**: SPM で Sparkle 追加、appcast 生成をリリースワークフローに組込み、EdDSA 鍵はオペレータ管理。
- **受け入れ基準(段階1)**: 新しいタグが存在するとき通知が出て、既知バージョン以下では出ない。
- **依存**: 段階1はなし。段階2は CP-02・CP-03。

### CP-09: Homebrew Cask 配布

- **背景/現状**: 現在の入手経路は GitHub Releases と copippe.com のみ。開発者層には `brew install --cask` が定番の導線。homebrew/cask 本体リポは知名度(notability)要件があるため、まず自前 tap から始める。
- **作業内容**:
  1. tap リポ用の cask 定義(`copippe.rb`)を作成: version / sha256 / url(GitHub Releases の ZIP)/ app stanza / zap stanza(設定・履歴ファイルのパス)。
  2. リリースワークフロー(CP-02)に cask の version/sha256 自動更新ステップを追加(tap リポへの PR 作成)。
  3. README / README-ja / LP に brew インストール手順を追記。
- **オペレータ作業**: tap 用リポ(例 `<owner>/homebrew-tap`)の作成と、ワークフローから tap へ push するための token 設定。
- **受け入れ基準**: `brew install --cask <owner>/tap/copippe` でインストールでき、`brew uninstall --zap` で関連ファイルが掃除される。
- **依存**: CP-02。CP-03 完了後だと Gatekeeper 摩擦なしで体験が良い。

### CP-10: 検索ポップアップの UX 強化

- **背景/現状**: Ctrl+Option+V のフローティング検索(`PopupContentView.swift`)はインクリメンタル検索対応済み。伸びしろ: スニペットが検索対象か不明(要実装時確認)、マッチ箇所のハイライトなし、キーボードのみでの操作完結性が未検証。
- **作業内容**:
  1. 現状把握: 検索対象(履歴のみ/スニペット含む)と現在のキーボード操作(矢印移動・Enter 貼付・Esc 閉じ)を確認し、この文書に追記してから着手。
  2. スニペットを検索対象に追加(セクション分け表示)。
  3. マッチ文字列のハイライト表示。
  4. キーボード完結: 数字キーでの先頭 N 件即貼付(Maccy 等の定番操作)を検討・実装。
  5. テスト: 検索フィルタリングロジックをビューから分離してユニットテスト化。
- **受け入れ基準**: マウスなしで「呼び出し→絞り込み→貼付」が完結する。
- **依存**: なし。

## P2: 中期改善

### CP-11: 除外アプリ設定

- **背景**: 特定アプリ(端末エミュレータ、VNC、パスワードマネージャ全般)からのコピーを履歴に記録したくないニーズ。CP-04 はアプリ側の自己申告(ConcealedType)頼みだが、こちらはユーザー指定の deny リスト。
- **作業内容**: コピー元アプリの bundle identifier を `NSPasteboard` 変化時にフロントモストアプリから推定して記録し、設定画面で除外アプリを追加・削除できるようにする。`ClipboardMonitor` の取り込み判定に組み込む。除外判定のユニットテスト追加。
- **受け入れ基準**: 除外指定したアプリからのコピーが履歴に入らない。
- **依存**: CP-04(同じ判定パイプラインに実装するため)。

### CP-12: アプリ内 UI のローカライズ(EN/JA)

- **背景/現状**: `.strings` / `.xcstrings` が存在せず、アプリ内文字列はハードコード。README・ユーザーガイドは EN/JA 両方あるのにアプリ UI は単一言語で、体験が不整合。
- **作業内容**: String Catalog(`.xcstrings`)を導入し、UI 文字列(メニュー、設定画面、ポップアップ、通知)を抽出して EN/JA を整備。以後の文字列追加ルールを AGENTS.md に追記。
- **受け入れ基準**: システム言語 EN/JA の双方で全 UI が自然に表示される。
- **依存**: CP-06/07/10 など UI 変更タスクの後に実施すると手戻りが少ない。

### CP-13: チェンジログの整備と LP への掲載

- **背景/現状**: リリースノートは GitHub Releases にのみ存在。LP(copippe.com)に更新履歴がなく、既存ユーザーが変化を追えない。
- **作業内容**: ルートに `CHANGELOG.md`(Keep a Changelog 形式)を新設し v0.1〜v0.3 を遡って記載。`docs/` に changelog ページ(EN/JA)を追加して LP からリンク。リリースワークフロー(CP-02)のリリースノート生成と整合させる。
- **受け入れ基準**: 次回リリース時に CHANGELOG → Release notes → LP の3点が同一内容で更新される手順が確立している。
- **依存**: CP-02。

## Codex への補足

- すべての変更で既存テストを壊さないこと。テストは実データ非破壊の隔離(`TestSupport.swift`)を踏襲する。
- UI 追加時は既存の SwiftUI + AppKit ハイブリッド構成(MenuBarExtra / NSPanel)の流儀に合わせる。
- 永続化フォーマット(`Persistence.swift` の JSON)を変更するタスク(CP-07 等)は、必ず旧フォーマットからの読み込み互換テストを追加する。
- コミットは Conventional Commits(`feat:` / `fix:` / `docs:` / `ci:` / `chore:`)。push・Release publish・タグ作成はユーザー確認を取ってから。
